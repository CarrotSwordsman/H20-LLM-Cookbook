# H20 vs H100 vs H200: what changes for LLM inference

H20 is part of NVIDIA's Hopper line (sm_90), but it is intentionally
de-tuned for the regulated-export market. It keeps Hopper's HBM3 memory
subsystem largely intact while cutting compute throughput by roughly **6.7×**
on bf16/fp8 dense GEMMs. This produces an unusual roofline that off-the-shelf
H100/H200 tuned kernel configs do not always hit.

## Spec sheet (SXM5, dense, no sparsity)

| Spec                          | H100 | H200 | **H20** |
|-------------------------------|-----:|-----:|--------:|
| Compute capability            | sm_90 | sm_90 | sm_90 |
| SM count                      |  132 |  132 |   **78** |
| BF16 / FP16 TFLOPS            |  989 |  989 |   **148** |
| FP8 TFLOPS                    | 1979 | 1979 |   **296** |
| INT8 TOPS                     | 1979 | 1979 |   **296** |
| HBM capacity                  | 80 GB | 141 GB | **96 GB** |
| HBM bandwidth                 | 3.35 TB/s | 4.8 TB/s | **4.0 TB/s** |
| L2 cache                      | 50 MB | 60 MB | **60 MB** |
| NVLink bandwidth (per GPU)    | 900 GB/s | 900 GB/s | **900 GB/s** |
| TDP                           | 700 W | 700 W | **400 W** |

Numbers are NVIDIA-published spec sheet values for the SXM5 form factor.

## Implication for kernel tuning

The compute-to-bandwidth ratio shifts dramatically:

| Card  | BF16 TFLOPS | HBM TB/s | Arithmetic intensity break-even |
|-------|------------:|---------:|--------------------------------:|
| H100  |        989  |    3.35  |                  **~295 op/B**  |
| H200  |        989  |    4.8   |                  **~206 op/B**  |
| H20   |        148  |    4.0   |                   **~37 op/B**  |

In a fused-MoE kernel that's bf16-compute and weight-streaming, this means:

- **Compute-bound regimes are much narrower on H20.** Even modestly large
  batches quickly become memory-bound, so H100's "go bigger to amortize" tile
  choices over-allocate registers and SMEM without help.
- **Smaller `BLOCK_SIZE_M` often wins.** Less work per stage means more
  overlap between MMA and HBM loads, which matters more when MMA is slow.
- **`num_stages = 4` or `5` is more frequently optimal**, because pipeline
  depth has to mask longer relative compute time.

This is exactly why H100/H200 configs perform sub-optimally when blindly
re-used on H20 — and why per-shape H20 tuning gives the speedups documented
in this cookbook.

## Implication for inference deployment

- **Memory tier**: closer to H100 (80 GB) than to H200 (141 GB), so the same
  context length / model size assumptions as H100 generally apply.
- **Bandwidth tier**: between H100 and H200, but closer to H200. Decode-phase
  throughput on bandwidth-bound workloads (Mixtral, large-K dense GEMMs) is
  **~20% faster than H100**.
- **Compute tier**: ~1/7 of H100. Prefill on long prompts and big-batch
  scenarios is markedly slower; **fp8 helps disproportionately on H20**
  because it doubles the compute-bound throughput while leaving the
  bandwidth-bound regime essentially unchanged.

In one-line summary: **H20 looks like an H200 from the memory side and an
A100-ish chip from the compute side**, and it benefits more than H100/H200
from per-shape kernel tuning.
